# Portfolio Specification

This document describes the portfolio as it exists now, then separates future Cloudflare Worker or SSR goals from current guarantees.

## Current Runtime

- The site is a Dioxus 0.7 web application.
- Routing is handled by a single `Route` enum in `src/main.rs`.
- The app is built as static web output with `dx build --platform web --release`.
- Cloudflare deployment uses Workers Static Assets through `wrangler.toml`.
- The deployed asset directory is `./target/dx/edgarcnp-dev/release/web/public`.
- SPA fallback is enabled with `not_found_handling = "single-page-application"`.
- There is no custom Worker request handler yet.
- There is no SSR guarantee yet.
- There are no server functions, persistent storage bindings, analytics scripts, or form submissions.

## Design Direction

The approved direction is Living Responsive Blueprint.

- The interface should feel precise, quiet, and engineered.
- Visual detail should support orientation, not become a forced "experience."
- Motion should be CSS-first, subtle, responsive, and respectful of `prefers-reduced-motion`.
- SVG animation is preferred when animation is needed.
- The site should remain fast, responsive, and useful across mobile, tablet, and desktop.
- The portfolio should communicate software engineering judgment through clear content, validated data, and reliable interaction patterns.

## Stack

- Rust stable.
- Dioxus 0.7 with router support.
- Tailwind CSS through root `tailwind.css`.
- Generated browser stylesheet at `assets/tailwind.css`.
- Local TOML and Markdown content under `content/`.
- `build.rs` generates the compile-time project/writing source manifest from the content directories.
- Cloudflare Wrangler for deployment and Cloudflare-style preview.
- Cloudflare Static Assets `_headers` file for deployment security headers.

Do not use removed Dioxus APIs such as `cx`, `Scope`, or `use_state`.

## Content Model

Content is bundled at compile time with `include_str!`, parsed with `toml` / `serde`, validated in `src/data/`, and then exposed to UI views through the data facade.

The source manifest for `content/projects/*.md` and `content/writing/*.md` is generated by `build.rs`. Adding a new Markdown project or writing file should not require editing Rust source registration arrays.

### Profile

- `name`
- `role`
- `summary`
- `email`
- `availability`

### Contact Links

- `label`
- `kind`
- `href`
- `detail`
- `external`

Contact kind is typed. Contact links are static only. There is no contact form and no visitor-message collection.

### Capabilities

- `label`
- `title`
- `description`

### Projects

Projects live as Markdown files with TOML frontmatter.

Frontmatter fields:

- `title`
- `slug`
- `summary`
- `year`
- `uploaded`
- `status`
- `technologies`
- `featured`
- `pinned`
- optional `links`

Project pages render title, status/year/links, and one Markdown-backed details panel. They do not render separate role, problem, constraints, approach, outcome, or stack sections.

Project cards are sorted by pinned state first, then upload date.

### Writing

Writing posts live as Markdown files with TOML frontmatter.

Frontmatter fields:

- `title`
- `slug`
- `summary`
- `published`
- `updated`
- `tags`

## Data Safety

- Content files are treated as data, not executable code.
- Content validation fails closed during lazy static content loading.
- Slugs must be URL-safe.
- Dates must be valid ISO dates.
- Project statuses must be one of the typed allowed statuses.
- Contact kinds must be one of the typed allowed contact kinds.
- Structured links allow only safe `https`, `mailto`, or root-relative URLs.
- External structured links must be HTTPS.
- Markdown raw HTML is escaped before it can reach `dangerous_inner_html`.
- Markdown link destinations are validated before rendering.
- Markdown images are allowed only for safe local `/assets/` image paths.

## UI Requirements

### Global Layout

- Use semantic landmarks: `header`, `nav`, `main`, and `footer`.
- Keep navigation visible and CSS-only.
- Expose active navigation state.
- Keep copy concise and scannable.

### Home

- Introduce name, role, summary, availability, and runtime/interface metadata.
- Show primary action to Projects and secondary action to Contact.
- Show featured projects.
- Show capabilities from structured content.
- Show latest writing.
- Show static contact paths.

### Projects

- Project cards should be full-card links.
- Pinned projects should have a visible indicator.
- Cards should show title, summary, year, status, technologies, and pinned state.

### Project Detail

- Render a title/header component.
- Render status and year.
- Render project links when present.
- Render one large Markdown-backed project details component.

### Writing

- Writing cards should be full-card links.
- Writing detail should render safe Markdown.
- Show published and updated dates.
- Show tags.

### Contact

- Render only static contact methods.
- Use safe link attributes for external links.
- Do not add a form unless the project explicitly changes direction.

### Not Found

- Show a clear 404 state and route users back to primary sections.
- The requested path is intentionally displayed for debugging because Dioxus escapes text output.

## Responsive Requirements

- Mobile: single column, visible wrapped navigation, no hover-only interactions.
- Tablet: use two-column layouts where content benefits from comparison.
- Desktop: constrained text widths, multi-column project grids where useful, persistent top navigation.
- Layout changes should feel stable and avoid text overlap.
- Dynamic responsive behavior should use CSS and SVG/CSS motion, not custom JavaScript.

## Accessibility

- All interactive elements must be keyboard reachable.
- Focus styles must be visible.
- Touch targets should be comfortable on mobile.
- Color cannot be the only state signal.
- Decorative blueprint artifacts should not add noise for assistive technology.
- Reduced-motion preferences must disable nonessential motion.
- Markdown-rendered content must remain readable and navigable.

## Motion

- Prefer CSS transitions and SVG animation.
- Avoid custom JavaScript animation.
- Animate cheap properties such as opacity, transform, color, background-color, and border-color.
- Avoid layout-triggering motion.
- Keep durations short.
- Respect `prefers-reduced-motion`.
- Large animation frameworks are allowed only if they respect reduced motion and provide clear value.

## Security Headers

Deployment headers are authored in `cloudflare/_headers` and copied into the generated public asset directory before Wrangler preview/deploy.

Current header categories:

- Content Security Policy
- Referrer Policy
- Permissions Policy
- Frame protections
- Content-type hardening
- Cross-origin isolation-adjacent protections that are compatible with the static app

The CSP must remain compatible with the generated Dioxus WebAssembly bootstrap.

## Verification

Canonical local checks:

```bash
cargo fmt --check
cargo check
cargo test
cargo clippy --all-targets --no-default-features --features web -- -D warnings
dx check
dx build --platform web --release
git diff --check
```

Cloudflare preparation:

```bash
npm run cloudflare:prepare
```

Cloudflare preview:

```bash
npm run wrangler:dev
```

Direct Cargo/Dioxus commands are the source of truth for app verification. npm scripts are retained for Wrangler/deployment convenience.

## Current Non-Goals

- No contact form.
- No analytics scripts.
- No theme toggle.
- No light mode.
- No database.
- No custom Worker request handler.
- No SSR guarantee.
- No route-transition framework.
- No JavaScript-only interaction requirement.

## Future Goals

Future work may add a custom Cloudflare Worker boundary, SSR, server functions, or storage bindings, but those changes must be designed and verified separately.

Before claiming Worker/SSR behavior:

- Add a clear Worker-specific boundary.
- Verify Cloudflare runtime behavior through Wrangler.
- Revisit CSP and headers against the actual runtime.
- Confirm generated bundles and assets stay within Cloudflare limits.
- Keep secrets and privileged bindings out of client bundles.
